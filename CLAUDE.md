## Overview

Wedding RSVP web app where each invitee receives a unique magic link containing a UUID token. On load the app looks up the invitee in Supabase and personalises the entire experience. The page is a single continuous scroll with a Three.js animated wedding cake, a Google Maps ceremony/reception reveal, and an RSVP form that saves responses back to Supabase.

**Type:** Web app
**Template:** next-supabase
**Stage:** See docs/PROJECT_STAGE.md

## Goal

Build a fully personalised, single-page wedding RSVP experience. Invitees arrive via a magic link (`/rsvp?token=<uuid>`), see a personalised welcome based on their guest type (Aus / Pom / Kiwi), scroll through an animated wedding cake destruction sequence, view ceremony and reception details with an animated map route reveal, and submit (or update) their RSVP at any time.

## Stack

**Base template:** next-supabase

**Add-ons / integrations:**

| Layer | Technology |
|-------|-----------|
| Frontend | Next.js (App Router), deployed on Vercel |
| Backend / DB | Supabase (Postgres + service-role client for all writes) |
| 3D / WebGL | Three.js via `@react-three/fiber` and `@react-three/drei` |
| Map | Google Maps JavaScript API (Polyline, custom markers) |
| Scroll animation | GSAP ScrollTrigger (drives Three.js timeline and map reveal) |
| Styling | Tailwind CSS |

## Testing standard

**Build passing is not the same as tested.**

When summarising any implementation chunk, always include a **"Testing performed"** section:

```
Testing performed
- Build/type checks:
    - [exact commands run]
- Manual testing:
    - [exact behaviours verified]
- Automated tests:
    - [tests added or run, or "none"]
- Limitations:
    - [what was not tested]
```

Rules:

1. Never say "tested" without specifying what kind and what was verified
2. Build checks confirm compilation only — they do not prove the feature works
3. If manual testing was not performed, say so explicitly
4. If no automated tests exist for this layer, say so — do not skip the section
5. For UI changes, check: happy path, validation, loading, success, error, disabled controls, light/dark mode
6. Add automated tests for pure logic/helpers where the test setup allows it

See `.claude/skills/testing-discipline/SKILL.md` for the full testing skill.

## Working principles

- Prefer clear, readable code over clever code
- Leave the code you touch better than you found it — but only what you touch
- When in doubt, ask — do not guess on architecture, scope, or user intent
- Keep documentation current: stale docs are worse than no docs
- Ship the smallest useful thing and iterate, rather than building everything at once
- If something feels wrong, it probably is — flag it before pressing on

## Approval gates

Stop and ask before proceeding when:

- Blocked on the same problem after 2 attempts
- The solution requires a different approach than originally planned
- About to simplify something to make it "work for now"
- About to drop scope without the user knowing

When asking, always present at least two options — never a single path:

```
I'm blocked on X. Here are the options:
1. [Option A] — faster but sacrifices Y
2. [Option B] — correct approach, takes longer
Which do you prefer?
```

**Working is not the same as production-ready.** Never treat them as equivalent without asking. Do not cut scope silently — propose it explicitly. If doing something properly takes longer, say so and confirm before proceeding — prefer correct over fast.

## Honest opposition

If the user's approach has a significant downside, say so — even if they seem committed. State disagreements directly ("I disagree because X"), not softened ("Great idea! One small thing…"). Show trade-offs even when confirming the user is right. Agreeing because it is easier is a failure mode.

## Time estimates

Estimate in Claude sessions ("1 focused session", "2–3 sessions depending on complexity"), not days or weeks, unless the user explicitly asks for calendar time.
