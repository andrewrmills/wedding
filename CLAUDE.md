## Testing standard

**Build passing is not the same as tested.**

When summarising any implementation chunk, always include a **"Testing performed"** section:

```
Testing performed
- Build/type checks:
    - [exact commands run]
- Manual testing:
    - [exact behaviours verified]
- Automated tests:
    - [tests added or run, or "none"]
- Limitations:
    - [what was not tested]
```

Rules:

1. Never say "tested" without specifying what kind and what was verified
2. Build checks confirm compilation only — they do not prove the feature works
3. If manual testing was not performed, say so explicitly
4. If no automated tests exist for this layer, say so — do not skip the section
5. For UI changes, check: happy path, validation, loading, success, error, disabled controls, light/dark mode
6. Add automated tests for pure logic/helpers where the test setup allows it

## UI — always-on rules

These apply any time you write or edit a UI component. No skill invocation needed.

1. **Check both modes.** Any coloured text must be readable in dark *and* light mode — not just the mode you are looking at.
2. **`text-*-100` through `text-*-400` are dark-mode-only pastels.** Never place them on a tinted background without confirming the variant is covered in the light-mode CSS override block. These classes look fine in dark mode and are invisible in light mode.
3. **Opacity-modifier classes bypass CSS overrides.** `text-red-500/70` generates a different selector than `text-red-500` — light-mode overrides will not catch it. Avoid opacity modifiers on text colour; use a darker base class instead.
4. **Selected and active states are the highest-risk spot.** Tinted highlight backgrounds make adjacent pale text nearly invisible. Use `text-slate-100` (which is overridden to dark in light mode) rather than a same-family pastel like `text-violet-200`.
5. **Check hover and focus state text colours too** — not just resting state. `hover:text-violet-300` is a pastel that can fail in light mode.
6. **Small text needs higher contrast.** The same colour that passes at `text-sm` can fail at `text-xs`. When in doubt, go one shade darker.
7. **Test at the largest zoom level the app supports.** Tight layouts and truncated text that look fine at 100% often break at 125–130%.

## Working principles

- Prefer clear, readable code over clever code
- Leave the code you touch better than you found it — but only what you touch
- When in doubt, ask — do not guess on architecture, scope, or user intent
- Keep documentation current: stale docs are worse than no docs
- Ship the smallest useful thing and iterate, rather than building everything at once
- If something feels wrong, it probably is — flag it before pressing on
